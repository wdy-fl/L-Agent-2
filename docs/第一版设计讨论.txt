✻ Cogitated for 13s
❯ 详细对比一下Hook和Middleware的区别                                            

⏺ Hook 和 Middleware 的核心区别可以一句话概括：

  ▎ Hook 是“在某个时点插入动作”，Middleware 
  ▎ 是“包住一段执行过程并控制它是否继续”。

  放到 L-Agent 的生命周期里，它们的抽象重心不同。

  ---
  1. Hook：阶段上的“插槽”

  Hook 更像事件监听器或扩展点。

  例如生命周期：

  before_agent → before_model → model_call → after_model → tool_call →
  after_tool → after_agent

  每个阶段可以注册多个 hook：

  before_model:
    - load_memory
    - load_skill_index
    - inject_workspace_guidance
    - check_context_budget

  每个 hook 做一件事：读取或修改 RunContext。

  伪代码大概是：

  for hook in hooks["before_model"]:
      context = hook(context)

  Hook 的特点

  优点：

  1. 简单直观
    - 很适合表达：“在模型调用前做某事”“工具调用后做某事”。
    - 对 F-Agent 现有逻辑改造成本低。
  2. 阶段感强
    - L-Agent 既然选择 lifecycle-first，hook 很自然。
    - before_model、after_tool 这些名字本身就是 hook 插槽。
  3. 适合能力挂载
    - 记忆召回可以是 before_model hook。
    - 技能索引注入可以是 before_model hook。
    - 工具结果裁剪可以是 after_tool hook。
    - 记忆同步可以是 after_agent hook。
    - 会话持久化可以是多个阶段 hook。
  4. 容易配置
  lifecycle:
    before_model:
      - memory.prefetch
      - skills.inject_index
      - context.check_budget

  缺点：

  1. 控制力弱
    - Hook 通常只“做事”，不负责完整控制调用链。
    - 它不太适合表达：“我包住 model_call，超时就中断，失败就重试”。
  2. 执行顺序容易变重要
    - 比如先注入 memory 还是先压缩 context，会影响结果。
    - 如果 hook 很多，阶段内部可能变成隐性的 mini pipeline。
  3. 错误处理容易分散
    - 每个 hook 失败后是继续、跳过、终止、降级？
    - 如果没有统一约定，会变乱。
  4. 跨阶段逻辑不自然
    - 比如一个预算控制逻辑，需要在 before_model 扣预算，在 after_model
  记录消耗，在异常时恢复状态。
    - 用 hook 可以做，但状态会散在多个 hook 里。

  ---
  2. Middleware：执行链上的“包裹器”

  Middleware 更像拦截器或洋葱模型。

  它不是简单挂在某个时点，而是包住一段核心执行：

  middleware_1(
    middleware_2(
      middleware_3(
        model_call
      )
    )
  )

  伪代码：

  def budget_middleware(context, next):
      if context.budget.exhausted:
          return stop()
      context.budget.consume()
      return next(context)

  或者：

  def timeout_middleware(context, next):
      with timeout(60):
          return next(context)

  Middleware 的特点

  优点：

  1. 控制力强
    - 可以决定是否继续执行 next()。
    - 可以提前返回、重试、捕获异常、包装结果。
  2. 适合横切能力
    - 预算控制
    - 超时控制
    - 重试
    - 日志追踪
    - 权限审批
    - tracing / metrics
    - sandbox / isolation
    - error boundary
  3. 天然适合包住高风险操作
    - model_call 可以被 timeout、retry、budget、logging middleware 包住。
    - tool_call 可以被 approval、sandbox、audit、result_truncate middleware
  包住。
  4. 前后逻辑集中
    - 一个 middleware 可以同时处理“调用前”和“调用后”。
    - 比 hook 更适合有状态的横切逻辑。

  缺点：

  1. 理解成本更高
    - 洋葱模型不如 hook 直观。
    - 新增 middleware 时要理解调用顺序和嵌套关系。
  2. 容易过度抽象
    - 如果所有东西都做成 middleware，Agent 生命周期会变成一条隐式调用链。
    - 反而削弱你想要的 before_agent → before_model → ... 显式阶段感。
  3. 配置不如 hook 直观
  model_call:
    middleware:
      - budget
      - timeout
      - retry
      - tracing
  3. 这可以配置，但用户要理解谁包谁、异常如何传播。
  4. 不适合普通能力挂载
    - 比如“读取 USER.md 注入上下文”用 middleware 就显得重。
    - “同步记忆”也不需要包住某段执行，hook 更自然。

  ---
  3. 放到 L-Agent 中的具体区别

  以 before_model → model_call → after_model 为例。

  Hook 风格

  before_model hooks:
    1. restore_session_context
    2. prefetch_memory
    3. inject_skill_index
    4. check_compression

  model_call:
    call_llm()

  after_model hooks:
    1. persist_assistant_message
    2. parse_tool_calls
    3. update_token_stats

  这表达的是：

  ▎ 到了某个生命周期阶段，顺序执行一组扩展动作。

  适合“阶段性准备”和“阶段性收尾”。

  ---
  Middleware 风格

  model_call = tracing(
    timeout(
      retry(
        budget(
          call_llm
        )
      )
    )
  )

  这表达的是：

  ▎ 对某个核心操作进行包装、控制和保护。

  适合“围绕一次调用建立控制边界”。

  ---
  5. 对 L-Agent 的建议

  我不建议 L-Agent 在宏观生命周期上直接采用纯 Middleware。

  因为你现在的目标是：

  ▎ 对 Agent 运行各环节进行抽象，实现各环节可插拔、可扩展的灵活配置。

  这里的关键词是 运行环节。所以主抽象应该保留清晰的阶段：

  before_agent
  before_model
  model_call
  after_model
  tool_call
  after_tool
  after_agent

  在这个前提下：

  推荐主设计

  生命周期阶段用 Hook / Pipeline Step，关键调用点内部用 Middleware。

  也就是：

  Lifecycle
    before_agent: hooks / steps
    before_model: hooks / steps

    model_call:
      middleware:
        - budget
        - timeout
        - retry
        - tracing
      handler:
        - llm_client.call

    after_model: hooks / steps

    tool_call:
      middleware:
        - approval
        - sandbox
        - audit
        - result_limit
      handler:
        - tool_registry.dispatch

    after_tool: hooks / steps
    after_agent: hooks / steps

  这样能兼顾两点：

  1. Agent 的宏观运行过程清楚
    - 生命周期还是显式的。
    - 一眼能看懂 Agent 从输入到回复经历了哪些阶段。
  2. 关键执行点有足够控制力
    - 模型调用、工具调用这种有成本、有风险、有失败可能的地方，用 middleware
  包住。
    - 记忆、技能、压缩、持久化这种阶段性能力，用 hook / step 挂载。

  ---
  6. 我对 L-Agent 的具体倾向

  如果只在 Hook 和 Middleware 之间选一个作为 L-Agent 的主抽象，我建议选：

  ▎ Hook 为主，Middleware 为辅。

  更准确地说：

  Lifecycle Stage = Hook Container
  Model Call / Tool Call = Middleware-wrapped Action

  原因是 L-Agent 不是 Web 框架，不需要把所有事情都变成 request
  middleware；它首先是一个 Agent 运行生命周期，应该让“阶段”成为一等公民。

  所以可以形成这样的概念分层：

  Agent
  └── Lifecycle
      ├── Stage
      │   ├── Hooks / Steps
      │   └── Action
      │       └── Middleware Chain
      └── RunContext

  这会比 F-Agent 当前的 AgentLoop 更优雅：

  - AgentLoop 不再亲自处理所有逻辑。
  - 它只负责推进 lifecycle。
  - 每个阶段挂什么能力由配置决定。
  - 模型调用和工具调用仍有强控制边界。
  - 未来扩展 MCP、子 Agent、任务系统、不同模型供应商时，不需要继续膨胀主循环。

  我的建议是：L-Agent 设计里不要把 Hook 和 Middleware 
  对立起来，而是明确它们的职责边界。Hook 表达生命周期扩展点，Middleware 
  表达对核心动作的控制链。

  我建议三者这样定义：

  1. Session：长期会话容器

  Session 是用户可感知的连续对话 / 工作上下文。

  它关心的是：

  - 这是哪一次会话
  - 历史消息是什么
  - 当前会话标题、统计、压缩状态是什么
  - 如何恢复上下文
  - 多轮 AgentRun 的结果如何沉淀

  可以理解为：

  一个 Session 包含多次用户输入，也就包含多次 AgentRun。

  ---
  2. AgentRun：一次 Agent 运行事务

  AgentRun 是 Agent 响应一次输入的完整运行过程。

  这个输入可以来自：

  - 用户在 CLI 输入的一句话
  - 定时任务触发的 prompt
  - 未来子 Agent 接收到的任务
  - 未来外部平台传入的消息

  AgentRun 关心的是：

  - 本次输入是什么
  - 本次运行属于哪个 session
  - 本次运行的 RunContext 是什么
  - 本次运行用了多少预算
  - 本次运行产生了哪些模型响应、工具调用、最终回复、错误、事件
  - 本次运行结束后如何提交到 session

  所以 AgentRun 是 执行边界 / 生命周期边界 / 临时状态边界。

  L-Agent 的七段生命周期，严格来说应该属于 AgentRun：

  Run 外层阶段

  ▎ 从某个稳定 checkpoint 复制一条“上下文视图”，以后新的消息写到新 branch，而旧 
  ▎ branch 保留不动。

  main:
    msg1 → msg2 → msg3 → msg4

  main:
    msg1 → msg2 → msg3 → msg4

  读取 fork_checkpoint 之前 main 的消息

  fork_checkpoint_id
  parent_branch_id
  base_message_cursor

  - 删除旧 branch 消息
  - 复制所有旧消息

  - 修改 checkpoint C 之前的历史

  设置一个新的 head 指针，而不是复制整棵树。

  ▎ 时间线上的一个可命名边界，记录“到这里为止，哪些上下文状态是稳定可恢复的”。

  session_id
  branch_id
  message_cursor
  run_id?
  iteration_index?
  kind
  stable

  - branch_id：属于哪条时间线
  - run_id：如果这个 checkpoint 是某次 AgentRun 产生的

  ▎ 用户可选择的稳定历史点。

  比如：

  assistant_message_committed
  tool_results_committed
  run_completed

  它们用于：

  - 展示历史点
  - 从某个点 rewind
  - 创建新 branch
  - 恢复模型上下文

  这类 checkpoint 必须稳定，不能在半个操作中间。

  ---
  3. 为什么还会有内部恢复 checkpoint

  这里我之前说得可能有点混。内部恢复 checkpoint 的目标不是用户 rewind，而是：

  ▎ 当 AgentRun 中断时，系统知道中断发生在哪个阶段，以及哪些副作用已经发生。

  它解决的是 运行中断后的诊断和安全恢复，不是用户主动回退。

  例如一次 AgentRun：

  before_agent
  before_model
  model_call
  after_model
  tool_call
  after_tool
  after_agent

  假设执行到工具阶段时程序崩了：

  model 返回 tool_calls
  tool_call 开始
  terminal 工具已经执行了
  还没把 tool result 写入消息表
  进程退出

  如果没有内部 checkpoint，恢复时你不知道：

  - tool_call 是否已经开始
  - 哪个 tool 已经执行
  - tool result 是否已经提交
  - 能不能安全重试
  - 是否应该提示用户“上次中断在工具执行中，可能有副作用”

  内部 checkpoint 的作用就是记录这些事实：

  tool_call_started
  tool_call_item_started: terminal abc
  tool_call_item_completed: terminal abc
  tool_results_committed

  这样恢复时可以做安全判断：

  如果 tool_call_started 但没有 tool_results_committed:
    标记上次 run 为 interrupted
    不自动重放工具
    提示用户上次中断在工具阶段

  ---
  4. 内部 checkpoint 和用户 checkpoint 的区别

  我建议不要把它们理解成两套完全不同的东西，而是同一个表里的不同可见性：

  Checkpoint
  ├── runtime checkpoint   # 系统内部用
  └── snapshot checkpoint  # 用户可见，可 resume/rewind

  区别是：

  维度: 主要用途
  runtime checkpoint: 崩溃恢复、诊断、安全判断
  snapshot checkpoint: resume / rewind / branch
  ────────────────────────────────────────
  维度: 用户是否可见
  runtime checkpoint: 默认不可见
  snapshot checkpoint: 可见
  ────────────────────────────────────────
  维度: 是否稳定
  runtime checkpoint: 不一定
  snapshot checkpoint: 必须稳定
  ────────────────────────────────────────
  维度: 粒度
  runtime checkpoint: 细，phase/action/tool item 级
  snapshot checkpoint: 粗，消息/结果/run 级
  ────────────────────────────────────────
  维度: 能否 rewind
  runtime checkpoint: 不能
  snapshot checkpoint: 能
  ────────────────────────────────────────
  维度: 典型例子
  runtime checkpoint: tool_call_started
  snapshot checkpoint: assistant_message_committed
  ────────────────────────────────────────
  维度: 关注点
  runtime checkpoint: “系统执行到哪里”
  snapshot checkpoint: “上下文可以恢复到哪里”

  ---
  5. 内部 checkpoint 有没有必要

  如果只做简单 resume / rewind，可以没有内部 checkpoint。

  但如果你希望 L-Agent 有更好的恢复、回退、安全性，我建议保留一个轻量内部
  checkpoint 机制，原因是：

  第一，避免重复执行有副作用工具

  工具调用是 Agent 里最危险的部分。

  如果上次 run 卡在 tool_call 中间，恢复时系统必须知道：

  这个工具是否已经执行过？
  结果是否已经写入？
  是否允许重放？

  否则自动恢复可能重复执行 write_file、terminal、mysql 等操作。

  第二，能标记 interrupted run

  没有 runtime checkpoint，只能知道“最后一条消息是什么”。

  有 runtime checkpoint，可以知道：

  run 12 interrupted at tool_call
  last completed stable checkpoint = model_response_committed
  pending tool calls = [...]

  这对 resume 很重要。

  第三，便于调试 Agent 运行过程

  你未来一定会想知道：

  - Agent 卡在哪个阶段
  - 哪个 step 失败
  - 哪个 middleware 中断
  - 为什么没有进入 after_agent

  runtime checkpoint 能作为运行事件日志的骨架。

  ---
  6. 但第一版可以怎么克制

  我建议第一版 checkpoint 不要复杂到“每个 step 都 checkpoint”。

  可以只做两类：

  用户可见 snapshot checkpoint

  创建在稳定上下文边界：

  user_message_committed
  assistant_message_committed
  tool_results_committed
  run_completed

  系统 runtime checkpoint

  只创建在高风险 action 边界：

  run_started
  model_call_started
  model_call_completed
  tool_call_started
  tool_call_completed
  run_failed
  run_interrupted

  暂时不要给每个普通 step 创建 checkpoint。

  这样足够支撑：

  - resume
  - rewind
  - branch
  - 工具中断安全判断
  - run 状态展示

  ---
  所以我建议我们目前先形成这个共同认知：

  Session 是会话容器。
  Branch 是 Session 内的上下文时间线。
  Rewind 不删除历史，而是从稳定 checkpoint fork 出新 branch。
  Checkpoint 是时间线边界。
  Snapshot checkpoint 给用户 resume / rewind。
  Runtime checkpoint 给系统判断 run
  是否完整、是否可安全恢复、是否可能重复副作用。

  下一步我们再讨论：哪些生命周期阶段应该产生 snapshot checkpoint，哪些阶段只产生
   runtime checkpoint。

  ▎ 根据当前 ReAct iteration 的上下文，检索可能相关的长期记忆 / 历史信息。

  memory.prefetch(original_user_input)

  一次 run 做一次，结果放入 base context。

  如果 memory retrieval 基于当前 ReAct 状态

  那它是 iteration-variant，应该放在 before_model：

  memory.retrieve(current_messages + latest_tool_results + current_goal)

  before_agent:

    - memory.refresh_for_iteration # 可选，默认关闭或仅在触发条件满足时执行

  before_agent:
    - memory.prefetch

  - 个人 CLI Agent 的大多数记忆召回由用户输入决定
  - 每轮检索会增加复杂度和成本
  - ReAct 中间态通常应通过 tool results 进入 message list，而不是反复查记忆
  - 真正需要动态召回时，模型可以通过 memory tool 显式查询

  更克制的说法应该是：

  ▎ 默认 memory retrieval 是 run 级 before_agent step；iteration 级动态 memory 
  ▎ refresh 是高级可选 step，不作为第一版默认。

  2. tool definitions filtering 是什么？

  我这里说的 tool definitions filtering 指的是：

  ▎ 每次 model_call 前，根据当前状态决定本轮暴露哪些工具 schema 给模型。

  例如：

  - 某些工具在当前模式下禁用
  - cron run 不允许交互式审批工具
  - rewind 后某些危险工具不允许自动出现
  - 预算不足时禁用 web_search
  - 当前任务不需要数据库工具
  - 用户配置禁用了 terminal
  - 子 Agent 只能使用只读工具
  - 已经进入“总结阶段”时不再暴露修改类工具

  也就是说，tool registry 里可能有很多工具，但本轮给模型的 tools
  可以是一个过滤后的列表。

  它为什么可能是 before_model？

  因为每轮状态可能变化：

  iteration 1:
    tools = read_file, list_files, web_search

  iteration 2:
    工具结果显示需要修改文件
    tools = read_file, write_file, terminal

  iteration 3:
    已经准备最终回答
    tools = none

  所以从最灵活的设计看，它属于 iteration-variant。

  但第一版是否需要？

  不一定。

  对于 L-Agent 第一版，如果目标还是 F-Agent 2.0，可以先做简单版：

  before_agent:
    - tool.snapshot_registry

  也就是本次 run 开始时确定可用工具集合。

  然后每轮 before_model 只是把这个工具集合放进当前 ModelRequest：

  before_model:
    - model_request.attach_tools

  不做复杂动态过滤。

  我现在更倾向的设计

  类似 memory，也分两层：

  before_agent:
    - tools.snapshot_available_tools    # 本 run 可用工具基线

  before_model:
    - tools.select_for_iteration        # 可选，本轮工具过滤

❯ 这个地方存在什么问题                                                          

⏺ 这里的问题在于：message.commit_user 和 checkpoint.create_user_snapshot
  不是两个普通 step，它们共同构成 rewind 的原子边界。

  如果处理不好，会出现几类不一致。

  1. 写入 user message 成功，但 checkpoint 失败

  流程：

  message.commit_user ✅
  checkpoint.create_user_snapshot ❌

  结果：

  - 会话里出现了一条用户消息
  - 但没有对应的 rewind checkpoint
  - 用户无法 rewind 到这条输入
  - branch 的可恢复边界不完整
  - 如果后续 run 中断，系统不知道这条 user message 是否是一个合法 run 起点

  这会破坏我们刚定的语义：

  ▎ 每个用户输入边界都是用户可见 snapshot checkpoint。

  ---
  2. checkpoint 成功，但 user message 失败

  流程：

  checkpoint.create_user_snapshot ✅
  message.commit_user ❌

  或者 checkpoint 先写了，但 message 没写成功。

  结果更糟：

  - checkpoint 指向一个不存在的 message_cursor
  - rewind 到这个 checkpoint 时无法恢复上下文
  - 新 branch 的 base_message_cursor 是坏的
  - resume/rewind 数据结构出现悬挂引用

  所以 checkpoint 必须依赖 user message 已成功提交。

  ---
  3. 两者之间 run 崩溃

  流程：

  message.commit_user ✅
  程序退出
  checkpoint.create_user_snapshot 未执行

  恢复时会看到：

  - 最后一条 message 是 user
  - 但没有 snapshot checkpoint
  - run 可能没有完成
  - active branch head 还在旧 run_completed

  这时系统要决定：

  这条 user message 是 orphan message？
  是否要补 checkpoint？
  是否要删除/隐藏它？
  是否标记 interrupted run？

  如果没有事务或恢复规则，resume 会很混乱。

  ---
  4. branch cursor 更新时机不清楚

  user_message_committed checkpoint 要记录：

  message_cursor = 当前 user message 的 sequence/id

  如果 message 写入和 checkpoint 写入不是同一个事务，就可能出现：

  - checkpoint cursor 指错
  - branch head 没更新
  - rewind 取不到正确的消息范围
  - 多线程/后台任务并发写入时 sequence 不稳定

  ---
  5. 用户 checkpoint 是否属于 run？

  还有一个语义问题：

  user_message_committed checkpoint

  它既是：

  - 用户可见 rewind 点
  - 本次 AgentRun 的起点
  - branch timeline 的一个稳定边界

  所以它应该关联：

  session_id
  branch_id
  run_id
  message_id / message_cursor

  如果 run.create、message.commit_user、checkpoint.create_user_snapshot
  三者不是一个明确事务，很容易出现：

  - message 有 run_id，但 checkpoint 没 run_id
  - checkpoint 有 run_id，但 run 不存在
  - run_started 了，但没有用户消息
  - 用户消息存在，但 run 失败后边界不清

  把这几个 step 合并成一个事务性 step，或者至少放进同一个 storage transaction：

  before_agent:
    - run.create_with_user_checkpoint

  它内部原子完成：

  1. create agent_run(status=running)
  2. append user message(branch_id, run_id, sequence)
  3. create checkpoint(

  4. mark run input committed

  before_agent failed

  → 不产生用户可见 checkpoint

  resume/rewind 语义。

  第一部分：核心概念与关系

  1. Agent

  Agent 是一个完整主体，不是某个 LLM 调用器。

  它包含：

  Agent
  ├── identity / guidance
  ├── AgentRunner
  ├── StepRegistry
  ├── Middleware
  ├── ToolRegistry
  ├── Memory interface
  ├── Skill interface
  ├── SessionStore
  └── Workspace

  Agent 的本质定义可以是：

  ▎ Agent 是一个围绕输入运行固定生命周期、通过模型决策和工具行动推进任务，并在会
  ▎ 话时间线上沉淀上下文状态的运行主体。

  ---
  2. Session

  Session 是长期会话容器，表示用户可感知的一次连续工作上下文。

  Session
  ├── session_id
  ├── title
  ├── active_branch_id
  ├── created_at / updated_at
  └── metadata / stats

  Session 不直接等于线性消息历史；它更像一个容器，内部可以有多条 branch。

  ---
  3. Branch

  Branch 是 Session 内的一条上下文时间线。

  Session
  └── Branch*
      ├── parent_branch_id
      ├── fork_checkpoint_id
      ├── base_message_cursor
      ├── resume_head
      └── messages / runs / checkpoints

  正常对话追加到 active branch。

  Rewind 时不删除旧历史，而是：

  从某个 user_message_committed checkpoint 创建新 branch

  ---
  4. AgentRun

  AgentRun 是 Agent 响应一次输入的完整运行事务。

  Session
  └── Branch
      └── AgentRun*

  一次 AgentRun 包含：

  - 一次输入
  - 一个 RunContext
  - 一个固定生命周期
  - 多轮 ReActIteration
  - 最终 completed / failed / interrupted 状态

  AgentRun 是生命周期边界。

  ---
  5. ReActIteration

  ReActIteration 是 AgentRun 内的一轮模型决策与可选工具行动。

  AgentRun
  └── ReActIteration*
      ├── before_model
      ├── model_call
      ├── after_model
      ├── before_tool?
      ├── tool_call?
      └── after_tool?

  如果模型没有 tool_calls，则本轮产生最终回复，AgentRun 进入 after_agent。

  ---
  6. Checkpoint

  Checkpoint 是 branch 时间线上的恢复边界，但分两类语义。

  Checkpoint
  ├── user snapshot checkpoint
  └── runtime checkpoint

  用户可见 checkpoint

  第一版只有：

  user_message_committed

  用于 rewind。

  Rewind 语义：

  rewind(checkpoint)
    → 从该 user message checkpoint 创建新 branch
    → 新 branch 上下文包含该 user message
    → 不包含它之后的 assistant/tool 结果

  内部 runtime checkpoint

  用于恢复诊断和避免重复副作用，例如：

  model_call_started
  model_call_completed
  assistant_message_committed
  tool_call_started
  tool_call_completed
  tool_results_committed
  run_completed
  run_failed
  run_interrupted

  它们默认不作为用户 rewind 点。

  ---
  7. Resume 与 Rewind

  resume:
    恢复 active branch 最新完整运行结果。

  也就是恢复到最近的 run_completed 对应状态，用于“接着聊”。

  rewind:
    回到某个 user_message_committed checkpoint，并创建新 branch。

  用于“从某次用户输入处重新开始”。

  ---
  8. 多 Agent

  多 Agent 不采用子 Agent 从属模型，而是 A2A 对等交互：

  Agent A ←A2A Protocol→ Agent B

  每个 Agent 都是完整 Agent。A2A
  后续作为协议能力接入，不进入第一版核心生命周期。

  第二部分：Lifecycle / Step / Action

  1. 基本规则

  L-Agent 第一版生命周期固定，不通过配置增删阶段。

  AgentRun
  ├── before_agent
  ├── ReActIteration*
  │   ├── before_model
  │   ├── model_call
  │   ├── after_model
  │   ├── before_tool?
  │   ├── tool_call?
  │   └── after_tool?
  └── after_agent

  其中：

  - before_agent / after_agent 是 run 级阶段，每次 AgentRun 只执行一次。
  - before_model / model_call / after_model 是 iteration 级阶段，每轮 ReAct
  都执行。
  - before_tool / tool_call / after_tool 只有模型返回 tool_calls 时执行。
  - Hook Phase 固定，Phase 内部的 Step 可注册、启用、禁用、调参、排序。
  - model_call 和 tool_call 是固定 Action + Middleware，不作为普通 step。

  ---
  2. RunContext 与 ModelContext

  一次 AgentRun 有一个全局可变 RunContext。

  RunContext
  ├── session / branch / run
  ├── input
  ├── base_model_context
  ├── iterations
  ├── current_model_request
  ├── current_model_response
  ├── current_tool_plan
  ├── current_tool_results
  ├── final_result
  ├── errors
  └── runtime state

  模型上下文拆为两层：

  BaseModelContext  # run 级不变部分
  ModelRequest      # iteration 级动态请求

  BaseModelContext 在 before_agent 构建：

  identity
  guidance
  workspace static context
  normalized user input metadata
  memory.prefetch result
  available tools snapshot
  model config

  ModelRequest 在每轮 before_model 构建：

  base_model_context
  visible messages
  compression/budget prepared context
  available tools
  current model params

  ---
  3. before_agent

  职责：

  ▎ 创建本次 AgentRun 的运行边界，提交用户输入 checkpoint，并准备本次 run 
  ▎ 内不变的 base model context。

  Steps：

  before_agent:
    - run.create
    - branch.resolve_active
    - context.initialize
    - input.normalize
    - message.commit_user
    - checkpoint.create_user_snapshot
    - base_context.load_static_parts
    - memory.prefetch
    - tools.snapshot_available_tools
    - budget.initialize

  说明：

  - message.commit_user 和 checkpoint.create_user_snapshot 共同形成用户可见
  rewind 边界。
  - memory.prefetch 只在 run 开始时做一次，基于本次输入召回 run 级记忆。
  - tools.snapshot_available_tools 只在 run 开始时确定本次 run 可用工具集合。
  - 不在第一版做每轮 memory refresh 或 dynamic tool filtering。

  ---
  4. before_model

  职责：

  ▎ 为当前 ReActIteration 构建一次具体的 ModelRequest。

  Steps：

  before_model:
    - iteration.create
    - messages.collect_visible
    - context.prepare_with_budget
    - model_request.compose

  说明：

  - messages.collect_visible 从当前 branch 时间线收集本轮可见 messages。
  - context.prepare_with_budget
  负责保证上下文符合模型窗口，但具体压缩策略后续单独设计。
  - model_request.compose 合并 base context、visible messages、tools snapshot
  和模型参数。
  - 预算是否允许调用模型不放这里，而放到 model_call middleware。

  ---
  5. model_call

  职责：

  ▎ 执行一次模型调用，并由 middleware 控制调用边界。

  结构：

  model_call:
    middleware:
      - budget.guard
      - timeout.guard
      - trace.record
    action:
      - llm.call

  AgentRunner 自动记录 runtime checkpoint：

  model_call_started
  model_call_completed
  model_call_failed

  说明：

  - budget.guard 控制 ReAct 最大轮数 / 模型调用预算。
  - timeout.guard 控制模型调用超时。
  - trace.record 记录耗时、usage、请求元信息。
  - llm.call 使用 ctx.current_model_request，写入 ctx.current_model_response。

  ---
  6. after_model

  职责：

  ▎ 把模型响应解释为 AgentRun 状态变化。

  Steps：

  after_model:
    - model.capture_response
    - message.commit_assistant
    - usage.update
    - result.detect_final_answer
    - tool.detect_requested

  内部 runtime checkpoint：

  assistant_message_committed

  说明：

  - 即使 assistant message 包含 tool_calls，也必须进入 message timeline。
  - 如果没有 tool_calls，则 result.detect_final_answer 设置
  ctx.final_result，本次 ReAct loop 准备结束。
  - 如果有 tool_calls，则 tool.detect_requested 标记进入工具段。

  ---
  7. before_tool

  职责：

  ▎ 把模型请求的 tool_calls 转换成可执行、可审计、可安全处理的 ToolPlan。

  Steps：

  before_tool:
    - tool_calls.extract
    - tool_calls.parse_arguments
    - tool_calls.validate_schema
    - tool_calls.resolve_tools
    - tool_plan.build_serial
    - approval.prepare_requests

  说明：

  - before_tool 做计划级解析与校验。
  - ToolDispatcher 仍做最终防御性校验。
  - 第一版工具执行保持全串行，不做 parallel_safe。
  - approval.prepare_requests 只准备审批上下文，不真正执行审批阻断。

  ---
  8. tool_call

  职责：

  ▎ 按 ToolPlan 串行执行工具，或在审批拒绝时生成拒绝类 tool result。

  结构：

  tool_call:
    middleware:
      - approval.guard
      - audit.record
      - result_limit.guard
    action:
      - tools.dispatch_serial

  AgentRunner 自动记录 runtime checkpoint：

  tool_call_started
  tool_call_completed
  tool_call_failed

  说明：

  - approval.guard 真正决定放行或拒绝。
  - 如果用户拒绝或策略拒绝，不终止 AgentRun，而是生成 denied tool result。
  - audit.record 记录工具执行审计。
  - result_limit.guard 统一限制工具结果大小。
  - tools.dispatch_serial 按 ToolPlan 顺序执行，并做最终防御校验。

  ---
  9. after_tool

  职责：

  ▎ 把工具执行结果提交为下一轮 ReAct 可见的 observation。

  Steps：

  after_tool:
    - tool_results.capture
    - message.commit_tool_results
    - checkpoint.record_tool_results_committed

  说明：

  - 每个 tool result 都写成 role=tool message，并保留 tool_call_id。
  - tool_results_committed 是内部 runtime checkpoint，不是用户 rewind 点。
  - 不在 after_tool 做上下文压缩；压缩/裁剪放到下一轮 before_model。

  ---
  10. after_agent

  职责：

  ▎ 结束本次 AgentRun，提交 run 级状态，并更新 resume 所需的 branch head。

  Steps：

  after_agent:
    - result.finalize
    - run.mark_terminal_state
    - checkpoint.record_run_terminal_state
    - branch.update_resume_head
    - stats.update_session
    - cleanup.release_runtime_state

  说明：

  - branch.update_resume_head 只在 run completed 时更新。
  - resume 恢复 active branch 最新完整运行结果。
  - 第一版不在 after_agent 自动执行 memory.sync。
  - 持久化记忆管理机制后续单独设计。
  - 上下文压缩策略也后续单独设计。

  第三部分：配置、注册与扩展边界

  1. 生命周期固定，Step 可扩展

  L-Agent 第一版不把生命周期本身做成可配置项。

  固定生命周期是 Agent 的语义骨架：

  before_agent
  before_model
  model_call
  after_model
  before_tool
  tool_call
  after_tool
  after_agent

  可扩展的是每个 Hook Phase 内部的 Step：

  Hook Phase = 固定生命周期阶段
  Step       = 阶段内部的可插拔能力模块

  ---
  2. Step 使用类接口

  Step 第一版采用类接口式：

  class Step:
      name: str
      phase: HookPhase

      def run(self, ctx: RunContext) -> None:
          ...

  例如：

  class MemoryPrefetchStep:
      name = "memory.prefetch"
      phase = "before_agent"

      def run(self, ctx: RunContext) -> None:
          ...

  第一版不做复杂 depends_on / config_schema / plugin 
  metadata，但保留后续扩展空间。

  ---
  3. Step Registry

  StepRegistry 负责：

  - 注册 Step
  - 按 phase 收集 Step
  - 按配置启用/禁用 Step
  - 按配置调整 Step 顺序
  - 为 AgentRunner 提供 phase -> steps

  概念上：

  StepRegistry
  ├── register(step)
  ├── get_steps(phase)
  └── configure(config)

  ---
  4. 配置策略

  你选择的是偏 A 的方式：

  ▎ 代码注册为主，配置只控制 Step 的行为。

  所以第一版配置不负责重写生命周期。

  配置只做：

  - Step enabled / disabled
  - Step 参数
  - Step 局部顺序

  例如：

  steps:
    memory.prefetch:
      enabled: true
      limit: 5

    tools.snapshot_available_tools:
      enabled: true

    context.prepare_with_budget:
      strategy: simple

  是否允许配置调整 step 顺序，可以分阶段：

  第一阶段：
    只支持 enabled 和参数。

  第二阶段：
    支持 order / before / after。

  第三阶段：
    支持外部插件注册新 step。

  ---
  5. Action 与 Middleware

  model_call 和 tool_call 不作为普通 Step。

  它们是固定 Action，由 middleware 包裹。

  Action:
    - llm.call
    - tools.dispatch_serial

  Middleware 第一版可以采用轻量接口：

  class Middleware:
      name: str
      target: ActionName

      def __call__(self, ctx: RunContext, next_call: Callable) -> Any:
          ...

  默认：

  model_call middleware:
    - budget.guard
    - timeout.guard
    - trace.record

  tool_call middleware:
    - approval.guard
    - audit.record
    - result_limit.guard

  配置也只控制 middleware 的启停和参数，不改变 action 本身。

  ---
  6. AgentRunner

  AgentRunner 是一次 AgentRun 的流程推进器。

  职责：

  - 创建并持有 RunContext
  - 执行 before_agent
  - 驱动 ReAct loop
  - 执行 before_model / model_call / after_model
  - 有 tool_calls 时执行 before_tool / tool_call / after_tool
  - 执行 after_agent
  - 自动记录 action 边界 runtime checkpoint
  - 根据 ctx.final_result / tool_calls / errors 判断流程走向

  AgentRunner 不负责具体业务：

  不直接做 memory
  不直接做 tool parsing
  不直接做 compression
  不直接做 persistence details

  这些都由 Step、Action、Middleware 或 Store 完成。

  ---
  7. 存储与时间线边界

  第一版核心存储概念：

  sessions
  branches
  agent_runs
  react_iterations
  messages
  checkpoints

  但当前设计阶段不展开事务细节。

  只保留语义要求：

  Session:
    会话容器

  Branch:
    上下文时间线

  AgentRun:
    一次输入到输出的运行事务

  ReActIteration:
    一轮模型决策和可选工具行动

  Message:
    conversation timeline 的事实记录

  Checkpoint:
    恢复边界

  关键规则：

  resume:
    使用 active branch 最新完整 run 状态。

  rewind:
    使用 user_message_committed checkpoint 创建新 branch。

  user_message_committed:
    第一版唯一用户可见 checkpoint。

  runtime checkpoints:
    用于内部恢复、诊断和副作用安全判断。

  ---
  8. 暂不进入第一版细节的内容

  - Step 插件生态
  - 工具并行调度
  - 复杂事务与恢复修复策略

       36 +├── identity / guidance                                       
       37 +├── AgentRunner                                               
       38 +├── StepRegistry                                              
       39 +├── Middleware                                                
       40 +├── ToolRegistry                                              
       41 +├── Memory interface                                          
       42 +├── Skill interface                                           
       43 +├── SessionStore                                              
       44 +└── Workspace                                                 
       45 +```                                                           
       46 +                                                              
       47 +定义：                                                        
       48 +                                                              
       49 +> Agent 是一个围绕输入运行固定生命周期、通过模型决策和工具行动
          +推进任务，并在会话时间线上沉淀上下文状态的运行主体。          
       50 +                                                              
       51 +### 2.2 Session                                               
       52 +                                                              
       53 +Session 是长期会话容器，表示用户可感知的一次连续工作上下文。  
       54 +                                                              
       55 +```text                                                       
       56 +Session                                                       
       57 +├── session_id                                                
       58 +├── title                                                     
       59 +├── active_branch_id                                          
       60 +├── created_at / updated_at                                   
       61 +└── metadata / stats                                          
       62 +```                                                           
       63 +                                                              
       64 +Session 不直接等于一条线性消息历史；它更像一个容器，内部可以有
          +多条 Branch。                                                 
       65 +                                                              
       66 +### 2.3 Branch                                                
       67 +                                                              
       68 +Branch 是 Session 内的一条上下文时间线。                      
       69 +                                                              
       70 +```text                                                       
       71 +Session                                                       
       72 +└── Branch*                                                   
       73 +    ├── parent_branch_id                                      
       74 +    ├── fork_checkpoint_id                                    
       75 +    ├── base_message_cursor                                   
       76 +    ├── resume_head                                           
       77 +    └── messages / runs / checkpoints                         
       78 +```                                                           
       79 +                                                              
       80 +正常对话追加到 active branch。Rewind 不删除旧历史，而是从某个 
          +`user_message_committed` checkpoint 创建新 branch。           
       81 +                                                              
       82 +### 2.4 AgentRun                                              
       83 +                                                              
       84 +AgentRun 是 Agent 响应一次输入的完整运行事务。                
       85 +                                                              
       86 +```text                                                       
       87 +Session                                                       
       88 +└── Branch                                                    
       89 +    └── AgentRun*                                             
       90 +```                                                           
       91 +                                                              
       92 +一次 AgentRun 包含：                                          
       93 +                                                              
       94 +- 一次输入                                                    
       95 +- 一个 RunContext                                             
       96 +- 一个固定生命周期                                            
       97 +- 多轮 ReActIteration                                         
       98 +- 最终 completed / failed / interrupted 状态                  
       99 +                                                              
      100 +AgentRun 是生命周期边界。                                     
      101 +                                                              
      102 +### 2.5 ReActIteration                                        
      103 +                                                              
      104 +ReActIteration 是 AgentRun 内的一轮模型决策与可选工具行动。   
      105 +                                                              
      106 +```text                                                       
      107 +AgentRun                                                      
      108 +└── ReActIteration*                                           
      109 +    ├── before_model                                          
      110 +    ├── model_call                                            
      111 +    ├── after_model                                           
      112 +    ├── before_tool?                                          
      113 +    ├── tool_call?                                            
      114 +    └── after_tool?                                           
      115 +```                                                           
      116 +                                                              
      117 +如果模型没有 tool_calls，则本轮产生最终回复，AgentRun 进入 `af
          +ter_agent`。                                                  
      118 +                                                              
      119 +### 2.6 Checkpoint                                            
      120 +                                                              
      121 +Checkpoint 是 Branch 时间线上的恢复边界，分为两类语义。       
      122 +                                                              
      123 +```text                                                       
      124 +Checkpoint                                                    
      125 +├── user snapshot checkpoint                                  
      126 +└── runtime checkpoint                                        
      127 +```                                                           
      128 +                                                              
      129 +第一版唯一用户可见 checkpoint：                               
      130 +                                                              
      131 +```text                                                       
      132 +user_message_committed                                        
      133 +```                                                           
      134 +                                                              
      135 +它用于 rewind。                                               
      136 +                                                              
      137 +Runtime checkpoint 默认不作为用户 rewind 点，只用于内部恢复、 
          +诊断和副作用安全判断，例如：                                  
      138 +                                                              
      139 +```text                                                       
      140 +model_call_started                                            
      141 +model_call_completed                                          
      142 +assistant_message_committed                                   
      143 +tool_call_started                                             
      144 +tool_call_completed                                           
      145 +tool_results_committed                                        
      146 +run_completed                                                 
      147 +run_failed                                                    
      148 +run_interrupted                                               
      149 +```                                                           
      150 +                                                              
      151 +### 2.7 Resume 与 Rewind                                      
      152 +                                                              
      153 +Resume：                                                      
      154 +                                                              
      155 +```text                                                       
      156 +恢复 active branch 最新完整运行结果。                         
      157 +```                                                           
      158 +                                                              
      159 +也就是恢复到最近的 `run_completed` 对应状态，用于“接着聊”。   
      160 +                                                              
      161 +Rewind：                                                      
      162 +                                                              
      163 +```text                                                       
      164 +回到某个 user_message_committed checkpoint，并创建新 branch。 
      165 +```                                                           
      166 +                                                              
      167 +Rewind 后，新 branch 上下文包含该 user message，但不包含它之后
          +的 assistant/tool 结果。                                      
      168 +                                                              
      169 +### 2.8 多 Agent                                              
      170 +                                                              
      171 +多 Agent 不采用子 Agent 从属模型，而是 A2A 对等交互：         
      172 +                                                              
      173 +```text                                                       
      174 +Agent A ←A2A Protocol→ Agent B                                
      175 +```                                                           
      176 +                                                              
      177 +每个 Agent 都是完整 Agent。A2A 后续作为协议能力接入，不进入第 
          +一版核心生命周期。                                            
      178 +                                                              
      179 +## 3. 固定八段生命周期                                        
      180 +                                                              
      181 +L-Agent 第一版生命周期固定，不通过配置增删阶段。              
      182 +                                                              
      183 +```text                                                       
      184 +AgentRun                                                      
      185 +├── before_agent                                              
      186 +├── ReActIteration*                                           
      187 +│   ├── before_model                                          
      188 +│   ├── model_call                                            
      189 +│   ├── after_model                                           
      190 +│   ├── before_tool?                                          
      191 +│   ├── tool_call?                                            
      192 +│   └── after_tool?                                           
      193 +└── after_agent                                               
      194 +```                                                           
      195 +                                                              
      196 +规则：                                                        
      197 +                                                              
      198 +- `before_agent` / `after_agent` 是 run 级阶段，每次 AgentRun 
          +只执行一次。                                                  
      199 +- `before_model` / `model_call` / `after_model` 是 iteration  
          +级阶段，每轮 ReAct 都执行。                                   
      200 +- `before_tool` / `tool_call` / `after_tool` 只有模型返回 tool
          +_calls 时执行。                                               
      201 +- Hook Phase 固定，Phase 内部的 Step 可注册、启用、禁用、调参 
          +、排序。                                                      
      202 +- `model_call` 和 `tool_call` 是固定 Action + Middleware，不作
          +为普通 Step。                                                 
      203 +                                                              
      204 +## 4. RunContext 与模型上下文                                 
      205 +                                                              
      206 +一次 AgentRun 有一个全局可变 RunContext。                     
      207 +                                                              
      208 +```text                                                       
      209 +RunContext                                                    
      210 +├── session / branch / run                                    
      211 +├── input                                                     
      212 +├── base_model_context                                        
      213 +├── iterations                                                
      214 +├── current_model_request                                     
      215 +├── current_model_response                                    
      216 +├── current_tool_plan                                         
      217 +├── current_tool_results                                      
      218 +├── final_result                                              
      219 +├── errors                                                    
      220 +└── runtime state                                             
      221 +```                                                           
      222 +                                                              
      223 +模型上下文拆为两层：                                          
      224 +                                                              
      225 +```text                                                       
      226 +BaseModelContext  # run 级不变部分                            
      227 +ModelRequest      # iteration 级动态请求                      
      228 +```                                                           
      229 +                                                              
      230 +`BaseModelContext` 在 `before_agent` 构建：                   
      231 +                                                              
      232 +```text                                                       
      233 +identity                                                      
      234 +guidance                                                      
      235 +workspace static context                                      
      236 +normalized user input metadata                                
      237 +memory.prefetch result                                        
      238 +available tools snapshot                                      
      239 +model config                                                  
      240 +```                                                           
      241 +                                                              
      242 +`ModelRequest` 在每轮 `before_model` 构建：                   
      243 +                                                              
      244 +```text                                                       
      245 +base_model_context                                            
      246 +visible messages                                              
      247 +compression/budget prepared context                           
      248 +available tools                                               
      249 +current model params                                          
      250 +```                                                           
      251 +                                                              
      252 +## 5. 各阶段 Step 分布                                        
      253 +                                                              
      254 +### 5.1 before_agent                                          
      255 +                                                              
      256 +职责：                                                        
      257 +                                                              
      258 +> 创建本次 AgentRun 的运行边界，提交用户输入 checkpoint，并准 
          +备本次 run 内不变的 base model context。                      
      259 +                                                              
      260 +Steps：                                                       
      261 +                                                              
      262 +```text                                                       
      263 +before_agent:                                                 
      264 +  - run.create                                                
      265 +  - branch.resolve_active                                     
      266 +  - context.initialize                                        
      267 +  - input.normalize                                           
      268 +  - message.commit_user                                       
      269 +  - checkpoint.create_user_snapshot                           
      270 +  - base_context.load_static_parts                            
      271 +  - memory.prefetch                                           
      272 +  - tools.snapshot_available_tools                            
      273 +  - budget.initialize                                         
      274 +```                                                           
      275 +                                                              
      276 +说明：                                                        
      277 +                                                              
      278 +- `message.commit_user` 和 `checkpoint.create_user_snapshot`  
          +共同形成用户可见 rewind 边界。                                
      279 +- `memory.prefetch` 只在 run 开始时做一次，基于本次输入召回 ru
          +n 级记忆。                                                    
      280 +- `tools.snapshot_available_tools` 只在 run 开始时确定本次 run
          + 可用工具集合。                                               
      281 +- 第一版不做每轮 memory refresh 或 dynamic tool filtering。   
      282 +                                                              
      283 +### 5.2 before_model                                          
      284 +                                                              
      285 +职责：                                                        
      286 +                                                              
      287 +> 为当前 ReActIteration 构建一次具体的 ModelRequest。         
      288 +                                                              
      289 +Steps：                                                       
      290 +                                                              
      291 +```text                                                       
      292 +before_model:                                                 
      293 +  - iteration.create                                          
      294 +  - messages.collect_visible                                  
      295 +  - context.prepare_with_budget                               
      296 +  - model_request.compose                                     
      297 +```                                                           
      298 +                                                              
      299 +说明：                                                        
      300 +                                                              
      301 +- `messages.collect_visible` 从当前 Branch 时间线收集本轮可见 
          +messages。                                                    
      302 +- `context.prepare_with_budget` 负责保证上下文符合模型窗口，但
          +具体压缩策略后续单独设计。                                    
      303 +- `model_request.compose` 合并 base context、visible messages 
          +、tools snapshot 和模型参数。                                 
      304 +- 预算是否允许调用模型不放这里，而放到 `model_call` middleware
          +。                                                            
      305 +                                                              
      306 +### 5.3 model_call                                            
      307 +                                                              
      308 +职责：                                                        
      309 +                                                              
      310 +> 执行一次模型调用，并由 middleware 控制调用边界。            
      311 +                                                              
      312 +结构：                                                        
      313 +                                                              
      314 +```text                                                       
      315 +model_call:                                                   
      316 +  middleware:                                                 
      317 +    - budget.guard                                            
      318 +    - timeout.guard                                           
      319 +    - trace.record                                            
      320 +  action:                                                     
      321 +    - llm.call                                                
      322 +```                                                           
      323 +                                                              
      324 +AgentRunner 自动记录 runtime checkpoint：                     
      325 +                                                              
      326 +```text                                                       
      327 +model_call_started                                            
      328 +model_call_completed                                          
      329 +model_call_failed                                             
      330 +```                                                           
      331 +                                                              
      332 +说明：                                                        
      333 +                                                              
      334 +- `budget.guard` 控制 ReAct 最大轮数 / 模型调用预算。         
      335 +- `timeout.guard` 控制模型调用超时。                          
      336 +- `trace.record` 记录耗时、usage、请求元信息。                
      337 +- `llm.call` 使用 `ctx.current_model_request`，写入 `ctx.curre
          +nt_model_response`。                                          
      338 +                                                              
      339 +### 5.4 after_model                                           
      340 +                                                              
      341 +职责：                                                        
      342 +                                                              
      343 +> 把模型响应解释为 AgentRun 状态变化。                        
      344 +                                                              
      345 +Steps：                                                       
      346 +                                                              
      347 +```text                                                       
      348 +after_model:                                                  
      349 +  - model.capture_response                                    
      350 +  - message.commit_assistant                                  
      351 +  - usage.update                                              
      352 +  - result.detect_final_answer                                
      353 +  - tool.detect_requested                                     
      354 +```                                                           
      355 +                                                              
      356 +内部 runtime checkpoint：                                     
      357 +                                                              
      358 +```text                                                       
      359 +assistant_message_committed                                   
      360 +```                                                           
      361 +                                                              
      362 +说明：                                                        
      363 +                                                              
      364 +- 即使 assistant message 包含 tool_calls，也必须进入 message t
          +imeline。                                                     
      365 +- 如果没有 tool_calls，则 `result.detect_final_answer` 设置 `c
          +tx.final_result`，本次 ReAct loop 准备结束。                  
      366 +- 如果有 tool_calls，则 `tool.detect_requested` 标记进入工具段
          +。                                                            
      367 +                                                              
      368 +### 5.5 before_tool                                           
      369 +                                                              
      370 +职责：                                                        
      371 +                                                              
      372 +> 把模型请求的 tool_calls 转换成可执行、可审计、可安全处理的 T
          +oolPlan。                                                     
      373 +                                                              
      374 +Steps：                                                       
      375 +                                                              
      376 +```text                                                       
      377 +before_tool:                                                  
      378 +  - tool_calls.extract                                        
      379 +  - tool_calls.parse_arguments                                
      380 +  - tool_calls.validate_schema                                
      381 +  - tool_calls.resolve_tools                                  
      382 +  - tool_plan.build_serial                                    
      383 +  - approval.prepare_requests                                 
      384 +```                                                           
      385 +                                                              
      386 +说明：                                                        
      387 +                                                              
      388 +- before_tool 做计划级解析与校验。                            
      389 +- ToolDispatcher 仍做最终防御性校验。                         
      390 +- 第一版工具执行保持全串行，不做 parallel_safe。              
      391 +- `approval.prepare_requests` 只准备审批上下文，不真正执行审批
          +阻断。                                                        
      392 +                                                              
      393 +### 5.6 tool_call                                             
      394 +                                                              
      395 +职责：                                                        
      396 +                                                              
      397 +> 按 ToolPlan 串行执行工具，或在审批拒绝时生成拒绝类 tool resu
          +lt。                                                          
      398 +                                                              
      399 +结构：                                                        
      400 +                                                              
      401 +```text                                                       
      402 +tool_call:                                                    
      403 +  middleware:                                                 
      404 +    - approval.guard                                          
      405 +    - audit.record                                            
      406 +    - result_limit.guard                                      
      407 +  action:                                                     
      408 +    - tools.dispatch_serial                                   
      409 +```                                                           
      410 +                                                              
      411 +AgentRunner 自动记录 runtime checkpoint：                     
      412 +                                                              
      413 +```text                                                       
      414 +tool_call_started                                             
      415 +tool_call_completed                                           
      416 +tool_call_failed                                              
      417 +```                                                           
      418 +                                                              
      419 +说明：                                                        
      420 +                                                              
      421 +- `approval.guard` 真正决定放行或拒绝。                       
      422 +- 如果用户拒绝或策略拒绝，不终止 AgentRun，而是生成 denied too
          +l result。                                                    
      423 +- `audit.record` 记录工具执行审计。                           
      424 +- `result_limit.guard` 统一限制工具结果大小。                 
      425 +- `tools.dispatch_serial` 按 ToolPlan 顺序执行，并做最终防御校
          +验。                                                          
      426 +                                                              
      427 +### 5.7 after_tool                                            
      428 +                                                              
      429 +职责：                                                        
      430 +                                                              
      431 +> 把工具执行结果提交为下一轮 ReAct 可见的 observation。       
      432 +                                                              
      433 +Steps：                                                       
      434 +                                                              
      435 +```text                                                       
      436 +after_tool:                                                   
      437 +  - tool_results.capture                                      
      438 +  - message.commit_tool_results                               
      439 +  - checkpoint.record_tool_results_committed                  
      440 +```                                                           
      441 +                                                              
      442 +说明：                                                        
      443 +                                                              
      444 +- 每个 tool result 都写成 `role=tool` message，并保留 `tool_ca
          +ll_id`。                                                      
      445 +- `tool_results_committed` 是内部 runtime checkpoint，不是用户
          + rewind 点。                                                  
      446 +- 不在 after_tool 做上下文压缩；压缩/裁剪放到下一轮 `before_mo
          +del`。                                                        
      447 +                                                              
      448 +### 5.8 after_agent                                           
      449 +                                                              
      450 +职责：                                                        
      451 +                                                              
      452 +> 结束本次 AgentRun，提交 run 级状态，并更新 resume 所需的 bra
          +nch head。                                                    
      453 +                                                              
      454 +Steps：                                                       
      455 +                                                              
      456 +```text                                                       
      457 +after_agent:                                                  
      458 +  - result.finalize                                           
      459 +  - run.mark_terminal_state                                   
      460 +  - checkpoint.record_run_terminal_state                      
      461 +  - branch.update_resume_head                                 
      462 +  - stats.update_session                                      
      463 +  - cleanup.release_runtime_state                             
      464 +```                                                           
      465 +                                                              
      466 +说明：                                                        
      467 +                                                              
      468 +- `branch.update_resume_head` 只在 run completed 时更新。     
      469 +- resume 恢复 active branch 最新完整运行结果。                
      470 +- 第一版不在 after_agent 自动执行 `memory.sync`。             
      471 +- 持久化记忆管理机制后续单独设计。                            
      472 +- 上下文压缩策略后续单独设计。                                
      473 +                                                              
      474 +## 6. Step、Registry 与配置                                   
      475 +                                                              
      476 +### 6.1 Step 类接口                                           
      477 +                                                              
      478 +Step 第一版采用类接口式：                                     
      479 +                                                              
      480 +```python                                                     
      481 +class Step:                                                   
      482 +    name: str                                                 
      483 +    phase: HookPhase                                          
      484 +                                                              
      485 +    def run(self, ctx: RunContext) -> None:                   
      486 +        ...                                                   
      487 +```                                                           
      488 +                                                              
      489 +第一版不做复杂 `depends_on / config_schema / plugin metadata` 
          +，但保留后续扩展空间。                                        
      490 +                                                              
      491 +### 6.2 StepRegistry                                          
      492 +                                                              
      493 +StepRegistry 负责：                                           
      494 +                                                              
      495 +```text                                                       
      496 +- 注册 Step                                                   
      497 +- 按 phase 收集 Step                                          
      498 +- 按配置启用/禁用 Step                                        
      499 +- 按配置调整 Step 顺序                                        
      500 +- 为 AgentRunner 提供 phase -> steps                          
      501 +```                                                           
      502 +                                                              
      503 +概念接口：                                                    
      504 +                                                              
      505 +```text                                                       
      506 +StepRegistry                                                  
      507 +├── register(step)                                            
      508 +├── get_steps(phase)                                          
      509 +└── configure(config)                                         
      510 +```                                                           
      511 +                                                              
      512 +### 6.3 配置策略                                              
      513 +                                                              
      514 +第一版采用：                                                  
      515 +                                                              
      516 +> 代码注册为主，配置只控制 Step 的行为。                      
      517 +                                                              
      518 +配置不负责重写生命周期，只做：                                
      519 +                                                              
      520 +```text                                                       
      521 +- Step enabled / disabled                                     
      522 +- Step 参数                                                   
      523 +- Step 局部顺序                                               
      524 +```                                                           
      525 +                                                              
      526 +示例：                                                        
      527 +                                                              
      528 +```yaml                                                       
      529 +steps:                                                        
      530 +  memory.prefetch:                                            
      531 +    enabled: true                                             
      532 +    limit: 5                                                  
      533 +                                                              
      534 +  tools.snapshot_available_tools:                             
      535 +    enabled: true                                             
      536 +                                                              
      537 +  context.prepare_with_budget:                                
      538 +    strategy: simple                                          
      539 +```                                                           
      540 +                                                              
      541 +配置能力可以分阶段推进：                                      
      542 +                                                              
      543 +```text                                                       
      544 +第一阶段：只支持 enabled 和参数。                             
      545 +第二阶段：支持 order / before / after。                       
      546 +第三阶段：支持外部插件注册新 step。                           
      547 +```                                                           
      548 +                                                              
      549 +## 7. Action 与 Middleware                                    
      550 +                                                              
      551 +`model_call` 和 `tool_call` 不作为普通 Step。它们是固定 Action
          +，由 Middleware 包裹。                                        
      552 +                                                              
      553 +```text                                                       
      554 +Action:                                                       
      555 +  - llm.call                                                  
      556 +  - tools.dispatch_serial                                     
      557 +```                                                           
      558 +                                                              
      559 +Middleware 第一版可以采用轻量接口：                           
      560 +                                                              
      561 +```python                                                     
      562 +class Middleware:                                             
      563 +    name: str                                                 
      564 +    target: ActionName                                        
      565 +                                                              
      566 +    def __call__(self, ctx: RunContext, next_call: Callable) -
          +> Any:                                                        
      567 +        ...                                                   
      568 +```                                                           
      569 +                                                              
      570 +默认：                                                        
      571 +                                                              
      572 +```text                                                       
      573 +model_call middleware:                                        
      574 +  - budget.guard                                              
      575 +  - timeout.guard                                             
      576 +  - trace.record                                              
      577 +                                                              
      578 +tool_call middleware:
